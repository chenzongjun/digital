# 配置报告模块开发指南

## 目标

使用 **React + JavaScript + Ant Design 4.24.5 + Less** 实现配置报告模块。模块通过跳转链接中的 `mode` 决定页面状态，支持新增、编辑、详情和审批，并在不同视口宽度下保持可用。

## 实现前先确认

- 从路由 query 中读取 `mode`，预期值为 `add`、`edit`、`detail`、`approval`。若项目已有枚举或值不同，复用项目定义；不要自行兼容未约定的别名。
- 接口地址、报告配置格式、富文本编辑器选型、审批权限和审批提交字段必须以现有项目约定为准。缺少约定时，将数据访问隔离在 service/adapter 中，并向需求方确认，不能编造接口协议。
- 先检查项目的路由方案、状态管理方案、表单封装、Less 变量和目录约定，优先沿用已有实现。

## 页面与响应式布局

页面由“左侧目录导航”和“中间报告项目区”组成，使用 flex 布局：

- 左侧为可展开/折叠的目录区域，报告区占用剩余空间，内容区必须设置 `min-width: 0`，避免窄屏时撑破布局。
- 优先使用 Ant Design 的 `Layout`、`Menu`、`Anchor`、`Button`、`Tooltip` 等组件；折叠按钮应始终可见且可键盘操作。
- 目录宽度、折叠状态和窄屏断点应集中管理。视口变窄（例如打开浏览器控制台）时，目录可折叠为图标或收起，报告区仍可阅读和编辑。
- 不要用固定的报告区宽度或依赖一次性读取 `window.innerWidth`。需要响应式状态时使用 `matchMedia` 或项目现有的响应式 Hook，并正确清理监听器。

## 导航联动

- 报告配置中的每个可见模块都必须有稳定且唯一的 `sectionId`；不要用数组下标作为长期 ID。
- 目录使用 `Menu` 展示模块，报告区使用 `Anchor` 或等价的锚点定位能力。点击目录后平滑定位到对应模块。
- 滚动导致报告区标题切换时，目录的选中项必须同步更新；菜单选中变化也必须反映当前可见模块。
- 将“当前模块 ID”作为单一状态来源，避免 `Menu`、`Anchor` 和滚动监听各自维护不一致的选中状态。
- 动态新增、删除或隐藏模块后，重新派生目录项和锚点；若当前模块被删除，选择相邻的可见模块或清空选中态。

## 组件拆分与数据边界

保持页面编排、数据状态和字段渲染分离，建议按职责拆分：

- `ReportConfigPage`：读取路由参数，加载配置/报告数据，编排页面状态与提交动作。
- `ReportModeProvider` 或页面级 Context：向深层组件提供只读的 `mode`、权限和报告基础信息；只共享确有跨层需求的数据，避免将所有局部状态放入 Context。
- `ReportDirectory`：只负责目录展示、折叠和定位请求，不直接修改表单值。
- `ReportSectionList` / `ReportSection`：根据配置渲染项目区和标题锚点。
- `FieldRenderer`：根据字段配置分发基础控件、可增删表格和富文本控件；未知字段类型应有明确开发期提示，不能静默丢失。
- `EditableTableField`、`RichTextField` 等复杂字段各自封装表单绑定、只读展示和校验逻辑。

将配置、初始数据、表单实例和保存动作通过明确 props 或页面级 Context 传递。子组件不得反向修改配置；表单值只能通过 Ant Design Form 的 API 更新。

## 表单、配置渲染与校验

- 使用 Ant Design `Form` 作为报告数据的唯一表单状态来源。字段配置至少应能表达 `name`、`label`、`type`、`required`、`rules`、`visible`、`readonly`、默认值和表格列配置。
- 每个动态字段的 `Form.Item name` 必须稳定、唯一，并与配置的字段路径对应。不要以展示标题作为字段 key。
- 普通控件优先映射到 Ant Design 基础组件，例如 `Input`、`InputNumber`、`Select`、`DatePicker`、`Radio`、`Checkbox`。
- 增删行表格使用 `Form.List` 管理行数据。行应有稳定业务 ID 或新建临时 ID；新增、删除、排序后仍要保证校验错误能定位到正确单元格。
- 富文本字段必须处理表单值转换、空内容校验和详情态展示。编辑器的具体库由项目现有依赖决定，不要无故引入新的编辑器依赖。
- 校验规则由字段配置和业务规则共同组成，使用 `Form.Item rules` / 自定义 validator 实现。提交前调用 `form.validateFields()`；失败时定位到第一个错误所属模块并同步更新目录选中态。
- 模块级校验错误数量可从 `form.getFieldsError()` 派生，用于在目录中显示错误状态；不要维护另一份独立的错误真相来源。
- 配置变化、模块增删或异步回填后，保证已不存在的字段值和校验状态不会误提交。

## 模式与权限

用集中函数或映射统一派生页面能力，避免在各组件散落 `mode === ...` 判断：

| 模式 | 预期能力 |
| --- | --- |
| `add` | 初始化默认值，可编辑、增删模块内容、保存/提交。 |
| `edit` | 加载已有数据，可编辑、增删模块内容、保存/提交。 |
| `detail` | 全部报告字段只读，不显示编辑和行操作。 |
| `approval` | 报告正文默认只读；仅开放已明确约定的审批控件和审批动作。 |

“审批态是否可编辑报告正文”和“哪些模块可增删”属于业务权限，必须由后端权限或明确配置决定，不能仅凭前端 `mode` 推断。

## 数据请求与异常状态

- 将加载配置、加载报告详情、保存草稿、提交报告和审批动作封装在独立 service 中；页面不要直接散落请求细节。
- 清晰处理 loading、空配置、加载失败、无权限和保存中状态，并避免重复提交。
- 异步请求完成前页面卸载或路由参数变化时，不应再更新过期页面状态。
- 提交 payload 由当前 Form 值和必要的报告元数据构成；不要把 UI 状态（目录折叠、当前锚点等）提交给后端。

## Less 与可维护性

- 样式使用 Less，并遵循项目已有命名、变量和主题方式。组件样式应限定在模块根选择器下，避免污染全局 Ant Design 样式。
- 只在实现判断不直观、配置字段含义或关键状态转换处写简短注释；重点注释 `mode` 能力映射、`sectionId` 规则和表单路径转换。
- 不做与需求无关的重构、抽象或依赖升级。每一处改动都应能对应到本模块需求。

## 验收标准

1. 四种 `mode` 均能从链接正确解析，并呈现相应的可编辑/只读/审批能力。
2. 目录点击可定位到对应报告模块；滚动报告区时目录高亮同步；模块动态增删后联动仍正确。
3. 在较窄视口下目录可折叠，报告区不溢出、不被遮挡，主要操作仍可访问。
4. 基础字段、可增删行表格和富文本均由配置正确渲染，并能回填、编辑或以只读方式展示。
5. 必填、自定义规则和表格单元格校验可阻止无效提交；提交失败会定位到第一个错误模块。
6. 覆盖模式能力映射、配置渲染、动态行增删和错误定位的关键单元测试；完成项目既有的 lint、测试和构建检查。
